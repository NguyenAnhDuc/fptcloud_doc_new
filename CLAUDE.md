# FPT Cloud Documentation Project

## Structure

Docusaurus site with 3 locale trees (currently separate docs plugin instances,
not Docusaurus i18n standard):

```
docs/        → English  (/docs/en/)
docs-vi/     → Tiếng Việt (/docs/vi/)
docs-ja/     → 日本語    (/docs/ja/)
```

All 3 trees share the **same folder hierarchy**, derived from the original
manual sidebar menu. Top-level folders are category wrappers:

- `billing/`
- `fpt-ai-factory/`, `fpt-appsec/`, `fpt-cloud-backup-and-dr/`,
  `fpt-cloud-server/`, `fpt-container/`, `fpt-data-platform/`,
  `fpt-database-engine/`, `fpt-devsecops-services/`, `fpt-integration/`,
  `fpt-monitoring/`, `fpt-network/`, `fpt-security/`, `fpt-storage/`
- `gpu-container/`, `gpu-virtual-machine-en/`, `managed-gpu-cluster-kubernetes/`,
  `managed-gpu-cluster-kubernetes-ja/`, `metal-cloud-ja/`
- `support/`
- `intro.md` (site root)

Each folder contains a **module** folder and each module contains sub-category
folders matching the old sidebar hierarchy.

### `_category_.json`

Every category folder has a `_category_.json` with localized label and position:

```json
{
  "label": "Quick Starts",
  "position": 2,
  "collapsible": true,
  "collapsed": true,
  "link": { "type": "generated-index" }
}
```

### Sidebar

`sidebars.js`, `sidebars-vi.js`, `sidebars-ja.js` all use a single
autogenerated entry:

```js
module.exports = {
  tutorialSidebar: [{ type: "autogenerated", dirName: "." }]
};
```

The menu structure comes entirely from the folder tree + `_category_.json`
files. Never hand-edit these sidebars.

## Adding a New Module

1. Decide the breadcrumb in the menu (e.g. `FPT Cloud Server > New Module > Quick Starts > ...`)
2. Create folders in all 3 locales:
   - `docs/fpt-cloud-server/new-module/`
   - `docs-vi/fpt-cloud-server/new-module/`
   - `docs-ja/fpt-cloud-server/new-module/`
3. Add `_category_.json` for the module folder in each locale (localized `label`)
4. Place `.md` files inside (optionally in sub-folders like `quick-starts/`, `tutorials/`)
5. Add `_category_.json` for each sub-folder with localized label + position
6. `npm run build` — autogenerate picks it up

No sidebar file edits needed.

## Frontmatter Conventions

Minimum for each `.md`:

```yaml
---
id: "short-slug"             # optional; Docusaurus infers from filename
title: "Page Title"           # required (or H1 in body)
sidebar_label: "Short Label"  # optional; defaults to title
sidebar_position: 3           # optional; controls order within folder
---
```

**Do not** use `slug: "/..."` overrides — URLs should be derived from folder
structure.

## Docsmith Integration

External tool: `docsmith` plugin (`/docsmith start <product>`) generates pure
markdown files into `docs/drafts/`. To integrate its output:

1. Run `/docsmith draft <product>` in a workspace
2. Copy resulting `.md` files from `docs/drafts/` into the appropriate folder
   in this project (e.g. `docs/fpt-cloud-server/new-module/quick-starts/`)
3. Add a `_category_.json` if creating a new sub-category
4. `npm run build` — done

Docsmith output is pure markdown with H1 headings, no frontmatter. Docusaurus
infers title from the H1 automatically. No script needed.

## Build

```bash
npm run start       # dev server on localhost:3000
npm run build       # production build (NODE_OPTIONS=8192 preset)
npm run serve       # serve the built site
```

Note: `package.json` build script is pre-configured with
`NODE_OPTIONS='--max-old-space-size=8192'` because the site is large
(~5000 docs per locale).

## Orphan Files

Files with `draft: true` frontmatter are excluded from production build
(kept in git for historical reference). 523 orphan files exist from the
mass migration — content not referenced by the original sidebar.

## Images

- Absolute paths only: `/img/migrated/<subfolder>/<img>.png`
- Do not use relative `./images/...` refs — they break when files move
- Images live in `static/img/migrated/`

## Style Guide

Read `fptcloud-docs-style-guide.md` at repo root before making content changes.

## Legacy: Module Review & Sync Workflow (historical)

The original module-by-module review workflow is no longer active since the
mass migration. Files are now organized to match the menu hierarchy; cross-
locale sync should happen at the file level (3 locales share the same folder
path for every doc).
